# Codex Implementation Workflow

## Purpose

`.github/workflows/implement.yml` is the implementation phase for one already
approved daily-audit proposal.

It sits after the future Approved Proposal Dispatcher and before normal Draft PR
review, CI, deployment approval, and production deployment.

This workflow does not create or replace the dispatcher.

## `workflow_dispatch` Contract

The workflow accepts exactly these inputs:

| Input | Required | Meaning |
| --- | --- | --- |
| `proposal_id` | Yes | Exact approved `proposal_id` to implement. |
| `approval_run_id` | Yes | Proposal Approval workflow run ID containing the `proposal-decision` artifact. |

The approval artifact identity is intentionally fixed by the workflow:

```text
artifact name: proposal-decision
required file: approved-proposal.json
```

The workflow does not accept `recommended_action`, `evidence`, full proposal
JSON, `ranked-proposals.json`, or `approval-decision.json` as dispatch inputs.
Proposal content is loaded only from `approved-proposal.json`.

## Validation Contract

`scripts/prepare-codex-implementation.mjs` independently validates the downloaded
`approved-proposal.json` before any Codex step runs.

The workflow stops on any validation failure:

- artifact file is missing
- JSON is malformed
- top-level `proposal_id` is missing or empty
- `decision` is not exactly `approve`
- top-level `source_run_id` is missing or empty
- requested `proposal_id` does not match the artifact `proposal_id`
- `proposal` object is missing
- `proposal.proposal_id` does not match the top-level `proposal_id`
- proposal fields needed for implementation and PR auditability are missing:
  `category`, `evidence`, scores, `overall_priority`, and
  `recommended_action`

The script writes only secret-free generated files under
`out/codex-implementation/`, which is ignored by Git.

## Branch Strategy

The implementation branch is generated by the validation script:

```text
codex/implement-<proposal-id-slug>-<implementation-workflow-run-id>-<proposal-id-hash>
```

The branch name is deterministic for the proposal ID and workflow run ID, while
the hash protects against proposal IDs that collapse to the same slug.

The workflow always checks out `main`, creates the dedicated implementation
branch, and never commits directly to `main`.

## Codex Execution

The workflow uses the officially documented Codex GitHub Action:

```yaml
uses: openai/codex-action@v1
with:
  openai-api-key: ${{ secrets.OPENAI_ACCESS_TOKEN }}
  prompt-file: out/codex-implementation/implementation-prompt.md
  sandbox: workspace-write
  safety-strategy: drop-sudo
  output-file: out/codex-implementation/codex-output.md
```

The repository already uses `OPENAI_ACCESS_TOKEN` as the OpenAI bearer token
secret for the Responses API proposal-generation stage. This workflow reuses
that existing secret convention instead of introducing a new secret name.

The generated prompt says the proposal has passed human approval, includes only
the validated approved proposal, requires repository inspection before editing,
forbids unrelated changes, requires existing validation, and forbids deployment,
merge, or marking the PR ready for review.

## Permissions

The workflow grants only the permissions needed for this phase:

| Permission | Level | Justification |
| --- | --- | --- |
| `actions` | `read` | Download the `proposal-decision` artifact from the approval workflow run. |
| `contents` | `write` | Create and push the dedicated implementation branch commit. |
| `pull-requests` | `write` | Create the Draft Pull Request targeting `main`. |

The workflow does not request `id-token: write`, deployment permissions,
packages permissions, environments bypass, or unrelated repository scopes.

## Draft PR Behavior

After Codex runs and repository validation passes, the workflow:

1. fails if Codex produced no repository changes
2. commits the repository changes on the dedicated implementation branch
3. pushes that branch
4. creates a Draft Pull Request targeting `main`

The Draft PR body includes:

- `proposal_id`
- approval workflow run ID
- source audit run ID
- implementation workflow run ID
- proposal category
- proposal evidence
- recommended action
- validation results

The workflow never marks the PR ready for review and never merges it.

## Later Dispatcher Requirement

The later Approved Proposal Dispatcher must call this workflow via
`workflow_dispatch` on `main` and send only:

```json
{
  "proposal_id": "<approved proposal_id>",
  "approval_run_id": "<Proposal Approval workflow run ID>"
}
```

The dispatcher must not pass proposal text, ranked proposals, approval decisions,
or implementation instructions directly.
